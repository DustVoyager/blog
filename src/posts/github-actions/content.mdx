# GitHub Actions로 시작하는 자동화 워크플로우

GitHub Actions는 GitHub에서 제공하는 CI/CD 및 자동화 플랫폼입니다. 코드 저장소에서 발생하는 다양한 이벤트를 기반으로 워크플로우를 자동으로 실행할 수 있어, 빌드, 테스트, 배포 등의 반복 작업을 효율적으로 관리할 수 있습니다.

## 1. GitHub Actions란?

GitHub Actions는 소프트웨어 개발 워크플로우를 자동화할 수 있는 강력한 도구입니다. 저장소에 푸시, PR 생성, 이슈 등록 등의 이벤트가 발생하면 미리 정의한 작업들이 자동으로 실행됩니다.

### 주요 특징

- **이벤트 기반 자동화**: Git 이벤트에 반응하여 워크플로우 실행
- **다양한 러너 환경**: Ubuntu, Windows, macOS 등 다양한 OS 지원
- **풍부한 마켓플레이스**: 수천 개의 사전 제작된 액션 사용 가능
- **무료 사용량 제공**: Public 저장소는 무제한, Private 저장소도 월 2,000분 제공

## 2. 핵심 개념

### 2.1 워크플로우 (Workflow)

자동화된 프로세스의 전체 흐름을 정의합니다. `.github/workflows/` 디렉토리에 YAML 파일로 작성합니다.

### 2.2 이벤트 (Event)

워크플로우를 트리거하는 저장소 활동입니다.

```yaml
# 예시: push, pull_request, schedule 등
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
```

### 2.3 잡 (Job)

워크플로우 내에서 실행되는 작업 단위입니다. 여러 스텝으로 구성되며 병렬 또는 순차 실행이 가능합니다.

### 2.4 스텝 (Step)

잡 내에서 실행되는 개별 작업입니다. 명령어 실행 또는 액션 사용이 가능합니다.

### 2.5 러너 (Runner)

워크플로우가 실행되는 서버 환경입니다. GitHub 호스팅 러너 또는 자체 호스팅 러너를 사용할 수 있습니다.

## 3. 기본 워크플로우 작성하기

### 3.1 Node.js 프로젝트 CI 워크플로우

```yaml
name: Node.js CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest

    strategy:
      matrix:
        node-version: [18.x, 20.x]

    steps:
      - name: 코드 체크아웃
        uses: actions/checkout@v4

      - name: Node.js 설정
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'

      - name: 의존성 설치
        run: npm ci

      - name: 린트 검사
        run: npm run lint

      - name: 테스트 실행
        run: npm test

      - name: 빌드
        run: npm run build
```

### 3.2 주요 구성 요소 설명

**name**: 워크플로우의 이름입니다. GitHub Actions 탭에 표시됩니다.

**on**: 워크플로우를 트리거할 이벤트를 지정합니다.

**jobs**: 실행할 작업들을 정의합니다.

**runs-on**: 잡이 실행될 러너 환경을 지정합니다.

**strategy.matrix**: 여러 버전/환경에서 테스트하기 위한 매트릭스 전략입니다.

**steps**: 순차적으로 실행될 작업 단계들입니다.

## 4. 실전 활용 예제

### 4.1 자동 배포 워크플로우

```yaml
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Node.js 설정
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: 의존성 설치
        run: npm ci

      - name: 프로덕션 빌드
        run: npm run build
        env:
          NODE_ENV: production

      - name: AWS 자격증명 설정
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ap-northeast-2

      - name: S3에 배포
        run: |
          aws s3 sync ./dist s3://my-bucket --delete
```

### 4.2 자동 릴리즈 생성

```yaml
name: Create Release

on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: 릴리즈 생성
        uses: actions/create-release@v1
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          tag_name: ${{ github.ref }}
          release_name: Release ${{ github.ref }}
          draft: false
          prerelease: false
```

### 4.3 코드 품질 검사

```yaml
name: Code Quality

on: [push, pull_request]

jobs:
  quality:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Node.js 설정
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: 의존성 설치
        run: npm ci

      - name: ESLint 검사
        run: npm run lint

      - name: Prettier 검사
        run: npm run format:check

      - name: TypeScript 타입 검사
        run: npm run type-check

      - name: 테스트 커버리지
        run: npm run test:coverage

      - name: 커버리지 리포트 업로드
        uses: codecov/codecov-action@v3
        with:
          files: ./coverage/coverage-final.json
```

## 5. 고급 기능

### 5.1 시크릿 관리

민감한 정보는 GitHub Secrets에 저장하여 안전하게 사용합니다.

```yaml
steps:
  - name: API 배포
    env:
      DATABASE_URL: ${{ secrets.DATABASE_URL }}
      API_KEY: ${{ secrets.API_KEY }}
    run: npm run deploy
```

### 5.2 조건부 실행

특정 조건에서만 스텝을 실행할 수 있습니다.

```yaml
- name: 프로덕션 배포
  if: github.ref == 'refs/heads/main'
  run: npm run deploy:prod

- name: 스테이징 배포
  if: github.ref == 'refs/heads/develop'
  run: npm run deploy:staging
```

### 5.3 캐싱으로 속도 개선

의존성 캐싱으로 워크플로우 실행 시간을 단축할 수 있습니다.

```yaml
- name: 의존성 캐시
  uses: actions/cache@v3
  with:
    path: ~/.npm
    key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-node-
```

### 5.4 아티팩트 저장

빌드 결과물을 저장하고 다운로드할 수 있습니다.

```yaml
- name: 빌드 결과물 업로드
  uses: actions/upload-artifact@v3
  with:
    name: build-output
    path: dist/

- name: 빌드 결과물 다운로드
  uses: actions/download-artifact@v3
  with:
    name: build-output
```

## 6. 모범 사례

### 6.1 워크플로우 최적화

- **필요한 경우에만 실행**: 경로 필터링 사용
  ```yaml
  on:
    push:
      paths:
        - 'src/**'
        - 'package.json'
  ```

- **병렬 실행 활용**: 독립적인 작업은 병렬로 실행
- **캐싱 적극 활용**: 빌드 시간 단축
- **매트릭스 전략 사용**: 여러 환경 동시 테스트

### 6.2 보안 고려사항

- **시크릿 사용**: 민감 정보는 절대 하드코딩 금지
- **최소 권한 원칙**: 필요한 권한만 부여
- **써드파티 액션 검증**: 신뢰할 수 있는 액션만 사용
- **의존성 업데이트**: Dependabot 활용

### 6.3 디버깅 팁

```yaml
- name: 디버그 정보 출력
  run: |
    echo "이벤트: ${{ github.event_name }}"
    echo "브랜치: ${{ github.ref }}"
    echo "커밋: ${{ github.sha }}"
    echo "액터: ${{ github.actor }}"
```

## 7. GitHub Actions의 장단점

### 장점

✅ **GitHub 네이티브 통합**: 저장소와 완벽한 통합
✅ **사용 편의성**: YAML 기반의 직관적인 설정
✅ **풍부한 에코시스템**: 수천 개의 사전 제작 액션
✅ **무료 사용량**: Public 저장소 무제한 사용
✅ **다양한 플랫폼 지원**: Linux, macOS, Windows 러너 제공

### 단점

❌ **실행 시간 제한**: 잡당 최대 6시간, 워크플로우당 최대 72시간
❌ **동시 실행 제한**: Free 플랜은 20개 동시 잡 제한
❌ **러너 성능**: GitHub 호스팅 러너는 고정 스펙
❌ **복잡한 워크플로우**: 복잡해질수록 관리 어려움
❌ **디버깅 제한**: 로컬 테스트가 어려움 (act 도구 활용 가능)

## 8. 마무리

GitHub Actions는 현대적인 소프트웨어 개발에서 필수적인 도구입니다. 간단한 CI 파이프라인부터 복잡한 배포 자동화까지 다양한 용도로 활용할 수 있습니다. 처음에는 간단한 워크플로우로 시작하여 점진적으로 기능을 확장해 나가는 것을 권장합니다.
